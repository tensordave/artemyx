# Changelog - v0.6.x

## v0.6.4 - 2026-03-18

### Bug Fixes

- **Rename datasets on in-memory and OPFS maps** (`layer-actions/rename.ts`, `controls/layer-control.ts`, `map.ts`): renamed layers/datasets disappeared and could not be shown again; two root causes: (1) the rename orchestrator captured GeoJSON data via MapLibre's `source.serialize()` which does not reliably return inline data across MapLibre versions, replaced with `getFeaturesAsGeoJSON()` from DuckDB (source of truth); (2) explicit config-defined layers (e.g. `walkshed-fill`) have user-chosen IDs that don't match the `dataset-` prefix, so `remapLayerId()` returned the same ID, causing `addLayer` to throw "layer already exists" and a subsequent cleanup loop to delete the freshly re-added layer; fix: explicit layers are now removed then re-added in place, and the cleanup loop skips layers already handled; also wired the `loadedDatasets` tracking set to `LayerToggleControl` so it stays in sync after rename (previously the old ID lingered and the new ID was never added)

- **Renamed dataset layer order** (`layer-actions/rename.ts`, `controls/layer-control.ts`): renaming a dataset with explicit config layers (e.g. `bikeways-line` in demo-config) caused it to paint on top of all other layers; root cause was `addLayersBefore()` removing explicit config layers and re-adding them via `map.addLayer(spec)` without a `beforeId`, which places the layer at the top of MapLibre's stack; fix captures the next layer above in the stack before removal and passes it as `beforeId` to preserve z-position; also added a `resyncLayerOrder()` call after rename succeeds to reconcile MapLibre's visual stack with the `layer_order` column in DuckDB

- **Buffer dissolve ignoring quadSegs** (`operations/buffer.ts`): dissolve mode produced identical curve smoothness regardless of the `quadSegs` parameter; root cause was the `ST_Simplify` tolerance in dissolve paths being a fixed percentage of buffer distance (5% initial), which collapsed the extra vertices that higher quadSegs values produce; replaced with a sagitta-based tolerance derived from the arc geometry: `0.5 * (1 - cos(pi / (2 * quadSegs)))` scales the tolerance to stay below the chord height of one buffer arc segment, preserving curve detail at any quadSegs setting; a floor of 0.5% of buffer distance ensures dissolve has enough simplification for topology repair at high quadSegs (where the sagitta is tiny); retry escalation multipliers adjusted from `[0.05, 0.15, 0.40]` (fixed fractions) to `[1x, 8x, 25x]` of the effective tolerance; non-dissolve buffer paths were unaffected

- **Operation Builder dropdown stale after rename** (`controls/layer-control.ts`, `controls/operation-builder-control.ts`, `map.ts`): renaming or deleting a dataset while the Operation Builder panel was open left the input dataset dropdowns showing stale names and IDs; root cause was that the rename and delete callbacks in `LayerToggleControl` refreshed the layer panel but had no way to notify the Operation Builder; added a public `refreshDatasets()` method on `OperationBuilderControl` (no-op when panel is closed) and a `setOperationBuilderControl()` setter on `LayerToggleControl` following the existing `setOnPanelOpen`/`setLoadedDatasets` pattern; wired in `map.ts` after both controls are constructed; called after rename success and after delete completion

- **Layer reorder silent failure for renamed datasets** (`db/datasets.ts`): clicking the reorder arrows on a renamed dataset (especially operation outputs created via the Operation Builder) did nothing; root cause was `swapLayerOrder()` using raw Arrow Proxy rows from `result.toArray()` for the `.find()` comparison (`r.id === idA`), which could fail `===` against plain JS strings; `getDatasets()` already converted Arrow rows to plain objects at line 331 but `swapLayerOrder` did not; fixed by mapping rows to plain objects with `String(r.id)` and `Number(r.layer_order)` before comparison, matching the convention used throughout the rest of `datasets.ts`

- **Startup log message ordering** (`db/core.ts`, `map.ts`, `controls/progress-control.ts`): progress control history showed init steps out of order, with "Initializing database..." appearing after "Database ready (OPFS)" and a duplicate "Database ready" entry; root cause was two independent logging paths racing: the main thread posted "Initializing database..." to history immediately, then the worker's init log (accumulated with earlier timestamps) was prepended via `unshift()` when the RPC resolved, pushing it visually after all worker entries; additionally, map.ts posted its own "Database ready" on top of the worker's "Database ready (OPFS)"; fixed by moving "Initializing database..." into the worker's `logInitStep()` sequence as the first entry so all DB startup messages flow through a single path with correct timestamps; removed the redundant main-thread "Database ready" message; added a timestamp sort in `injectHistory()` as a safety net against future ordering issues

### Config Editor

- **Edit mode alignment fix** (`controls/config-control.ts`, `styles/config-viewer.css`): the transparent textarea and syntax-highlighted overlay no longer drift out of sync on longer configs; root cause was the proportional scroll-sync formula (`scrollTop / tMax * hMax`) accumulating error when the textarea and highlight div had slightly different `scrollHeight` values due to subpixel rendering differences between element types; replaced with a single-scroller approach: the textarea auto-grows to its full content height (`overflow: hidden`, JS sets `height = scrollHeight` on each input), the highlight div fills it via `inset: 0`, and the body element is the sole scroll container; scroll position is preserved across edit/view transitions and body scroll is saved around the `height: auto` reset to prevent jump-to-top on input

### Config Generation

- **Visibility in generated config** (`config/generator.ts`, `config/types.ts`, `config/validators/datasets.ts`, `data-actions/load-config.ts`): generated configs now include `visible: false` on datasets and operation outputs whose layers were toggled off via the layer panel eye icon; on re-run, visibility state is written to DuckDB during config loading so the existing post-pipeline visibility sweep restores it; only emitted when false (visible is the default), keeping generated YAML clean

### Bug Fixes

- **TopologyException on large dissolve operations** (`operations/buffer.ts`, `operations/union.ts`): buffer dissolve and union dissolve on dense datasets (e.g., 67k water main linestrings) threw `TopologyException: found non-noded intersection` regardless of buffer distance or quadSegs; root cause is GEOS building a single sweep-line graph over all N geometries where adjacent buffer rings have near-coincident edges that cannot be resolved at floating-point precision; fixed with a two-level chunked union: features are assigned row numbers, grouped into chunks of 500, each chunk is unioned independently, then the chunk results are unioned in a final pass; reduces per-call GEOS complexity from N to ~500 and limits the non-coincident edge problem to locally-clustered features; a retry loop escalates ST_Simplify tolerance (5% to 15% to 40% of buffer distance) as a safety net if TopologyException still occurs; final fallback returns individual (non-dissolved) buffers with a warning rather than failing
- **Config layers paint reset on Run** (`layers/layers.ts`): explicit config layers (e.g. `circle-radius: 3`) were being overwritten with default style values (`circle-radius: 6`) when pressing Run Config; `restoreStoredPaint()` was applying the dataset's stored style to all layers for a source, including config-defined layers that have their own paint from YAML; now skips non-default layers (those not matching the `dataset-{id}-fill/line/point` naming convention) so explicit config paint properties are preserved; OPFS restore was unaffected because it doesn't re-run the config pipeline
- **Operation builder stale status** (`controls/operation-builder-control.ts`): status message (success/error) now clears when the user changes the operation type or modifies any parameter field; previously status only cleared on output name edit or dataset selection, so stale feedback lingered while the user configured the next operation
- **Progress control stale message hang** (`controls/progress-control.ts`, `db/worker.ts`): the progress control would hang on a stale "processing" spinner after operations completed (e.g. "Filtering..." or "Differencing..." stuck after success messages); two fixes applied: (1) worker now flushes all pending batched progress events before responding with the operation result, guaranteeing correct message ordering on the main thread; (2) progress control ignores incoming `processing` messages when idle is already scheduled (defense-in-depth against any remaining timing edge cases) -- stale messages still log to history but don't overwrite the displayed state or re-render the icon
- **Operation builder missing scheduleIdle** (`controls/operation-builder-control.ts`): `handleRun()` was the only loading path that never called `logger.scheduleIdle()` after completion; added `scheduleIdle(3000)` after success and `scheduleIdle(5000)` after error, matching the convention used by the config pipeline, URL loader, and file loader

## v0.6.0 - 2026-03-10

### Worker-based DuckDB Pipeline

- **Worker architecture** (`db/worker.ts`, `db/worker-types.ts`, `db/client.ts`): full data pipeline (fetch, format detection, parse, DuckDB insert, operation compute) moved into a dedicated module worker; typed discriminated union message protocol for all DB operations and pipeline RPCs; main-thread RPC client with monotonic request IDs, pending Promise map, and 120s timeout; event handler registration for progress/info/warn forwarding and CRS prompt round-trip (pauses worker pipeline while main thread shows dialog); `db.ts` facade re-exports switched to `client.ts` - all existing callers unchanged
- **Pipeline delegation** (`load-url.ts`, `load-file.ts`, `executor.ts`): URL loading delegates entire fetch/parse/insert pipeline to worker via `loadFromUrl()`; file loading transfers `ArrayBuffer` zero-copy via `loadFromBuffer()`; executor uses `executeOperationInWorker()` for compute then renders on main thread with `addOperationResultToMap()`; main thread reduced to pre-flight checks, MapLibre rendering, and UI
- **Module isolation** (`db/constants.ts`): pure constants, types, and localStorage helpers extracted into a standalone module with no DuckDB imports; prevents `core.ts` from being pulled onto the main thread via transitive imports

### Worker Integration Fixes

- **OPFS wipe-and-retry** (`db/core.ts`): two-attempt OPFS init loop; on first failure (e.g. corrupted config making SQL unparseable), deletes the OPFS file and creates a fresh worker for retry; previously the broken file persisted, creating a permanent failure loop
- **Init promise wiring** (`db/worker.ts`): worker init now calls `startInit()` + `await ensureInit()` instead of `initDB()` directly; the old path never set `initPromise`, causing every subsequent `getConnection()` to spawn a competing DuckDB instance
- **Arrow Proxy serialization** (`db/datasets.ts`): `getDatasets()` maps Arrow Proxy rows to plain objects before returning; Proxy objects cannot survive `postMessage` structured clone

### Performance

- **GeoJSON as Transferable buffer** (`db/features.ts`): replaced per-row `JSON.parse` with a single `string_agg` SQL query building the entire FeatureCollection inside DuckDB; encoded as `Uint8Array` and transferred zero-copy via `postMessage` Transferable; eliminates structured clone overhead and 100K+ JS `JSON.parse` calls
- **SQL-based bounds** (`db/features.ts`): `getDatasetBounds()` returns 4 numbers via `MIN(ST_XMin)`/`MAX(ST_XMax)` aggregation; replaces round-tripping full FeatureCollections through structured clone just to compute bounding boxes
- **`getDatasetById()`** (`db/datasets.ts`): targeted single-row query replacing `getDatasets()` full table scan + `.find()` patterns across OPFS restore paths and worker load pipelines; also fixes `getDatasets()[0]` bug where multi-dataset loads returned wrong metadata
- **Hidden dataset skip**: hidden datasets (source-only, no rendering) skip GeoJSON materialization entirely during OPFS restore and worker pipeline; uses metadata query instead
- **Parallelized loops** (`map.ts`): `getDatasetBounds` and `restoreLabelIfConfigured` loops run concurrently via `Promise.all()` instead of sequential awaits; consolidated duplicate `getDatasets()` calls in post-pipeline section

### In-Memory Mode Memory Management

- **VACUUM compaction** (`db/core.ts`): `vacuum()` wired through worker RPC; compacts DuckDB's buffer pool after deletions and operations; in-memory mode previously grew monotonically since DELETE only marked pages as reusable
- **Reduced peak memory** (`db/datasets.ts`, `data-actions/`): `loadGeoJSON()` uses `registerFileBuffer()` instead of `registerFileText()` to avoid holding both JS string and DuckDB copies; GeoJSON references nulled immediately after MapLibre source creation for earlier GC
- **MapLibre tile cache reduction** (`layers/sources.ts`): GeoJSON sources configured with `tolerance: 0.5` and `buffer: 64` (reduced from defaults) to shrink internal tile cache footprint
- **Page navigation teardown** (`map.ts`): `pagehide` handler calls `map.remove()` then `worker.terminate()` to release WebGL context, WASM heap, and sub-workers; BroadcastChannel stored and closed in `onRemove()` to prevent listener leaks

### Loader Response Decoupling

- **Portable loader signatures** (`loaders/types.ts`): `FormatLoader` interface refactored from `load(response: Response)` to `load(data: LoaderData)` where `LoaderData = string | object | ArrayBuffer`; loaders no longer depend on the browser `Response` API, enabling use from Node.js and Web Workers
- **Caller-side unwrapping** (`load-url.ts`, `load-file.ts`): `Response`/`File` unwrapping moved from loaders into data-actions callers; `dispatch()` now takes `(data, format, options)` instead of `(response, format, options)`

### Config Pipeline Fixes

- **Layer ordering fix** (`map.ts`): visibility restoration reused a stale dataset snapshot from before `setLayerOrders` wrote config-derived order to DB; replaced with a fresh `getDatasets()` call so `resyncLayerOrder` receives the correct order
- **fitBounds pipeline** (`map.ts`): moved from `loadDatasetsFromConfig` (before operations) to end of full pipeline so operation outputs are included in bounds; bounds computed via `getDatasetBounds()` inside the worker load pipeline and merged into a single envelope; replaces main-thread GeoJSON coordinate iteration; uses per-geometry min/max aggregation instead of `ST_Extent` which returned degenerate bounding boxes in DuckDB-WASM; `maxZoom: 17` cap added

### Idle CPU Reduction

- **Removed `backdrop-filter: blur()`** (`controls.css`, `progress.css`, `legend.css`, `context-menu.css`): 6 blur declarations removed; each forced GPU compositing over MapLibre's WebGL canvas every frame; replaced with higher background opacity; idle CPU dropped from 6-12% to 1-3%
- **Throttled mousemove handlers** (`popup.ts`, `scale-control.ts`): hover tooltip and coordinate display handlers gated behind `requestAnimationFrame` to cap at 60fps

### Bug Fixes

- **Union dissolve TopologyException** (`operations/union.ts`): replaced `ST_Simplify` with `ST_MakeValid` before `ST_Union_Agg` in dissolve mode; degenerate geometries caused `TopologyException` that simpler repair tricks couldn't fix
- **Legend collapsed by default** (`controls/legend-control.ts`): legend panel now defaults to collapsed on all screen sizes; previously expanded on desktop (>640px); user preference still persisted via localStorage

### Operation Compute/Render Split

- **Pure compute functions** (`operations/*.ts`): each operation exports `compute<Name>(connection, op, callbacks?)` with explicit DuckDB connection parameter and optional `ComputeCallbacks` for progress; returns `ComputeResult` with output metadata; no MapLibre imports in compute path
- **Shared render module** (`operations/render.ts`): `addOperationResultToMap()` extracted from `buffer.ts`; handles MapLibre source/layer creation for all 8 operations
- **Dual exports** (`operations/index.ts`): both `execute<Name>` (compute + render) and `compute<Name>` (pure SQL) exported; compute exports are the contract for Web Worker and future CLI
