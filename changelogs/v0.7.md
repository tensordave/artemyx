# Changelog - v0.7.x

## v0.7.2 - 2026-03-25

### Output Progress UX

- **Output status lifecycle** (`config/output-types.ts`): new `OutputStatus` type (`pending | generating | complete | error`) and `OutputProgressCallback` type for streaming per-output updates to the UI; `OutputResult` gains `status`, `statusMessage` (human-readable progress text), and `progress` (numeric 0-1 for determinate progress bars) fields
- **Worker progress plumbing** (`db/worker-types.ts`, `db/worker.ts`, `db/client.ts`, `db.ts`): `ProgressEvent` gains optional `progress` field (0-1); `postProgress()` accepts and forwards it; `WorkerEventHandler.onProgress` updated to include the numeric value; new secondary subscriber system via `addProgressListener()`/`removeProgressListener()` so controls can tap into worker progress events independently of the primary event handler used by ProgressControl
- **PMTiles deterministic progress** (`db/pmtiles-writer.ts`): `onProgress` callback updated to `(message, progress?)` signature; reports numeric progress at each pipeline stage: 0 (reading features), 0.05 (slicing), 0.1-0.9 (per-zoom-level tiling, scaled linearly across the zoom range), 0.95 (building archive)
- **Source existence pre-check** (`config/output-executor.ts`): new `checkSourcesExist()` queries DuckDB for all unique output source IDs before execution starts; returns `{ allExist, missing }` so the UI can show a clear message when the config hasn't been run yet; `executeOutputs()` callback changed to `OutputProgressCallback` that fires at each status transition with the output index and full result object
- **Live progress dropdown** (`controls/config-control.ts`): outputs dropdown opens immediately on click with all rows in `pending` state; rows update in-place via `updateOutputRow()` as each output transitions through generating/complete/error; PMTiles outputs get granular updates from a secondary worker progress listener that patches `statusMessage` and `progress` on the generating row; row rendering factored into `renderOutputRow()` with status-driven layout (spinner + message for pending/generating, error text for failures, size + download button for complete); "Download All (.zip)" appended after all outputs finish; dropdown positioning extracted to shared `positionOutputsDropdown()`
- **No-data guard** (`controls/config-control.ts`): clicking the outputs button when source datasets haven't been loaded shows a notice dropdown ("Run the config first to load data") listing each missing source ID in red, instead of silently failing or producing confusing errors
- **CSS** (`styles/config-viewer.css`): pending rows dimmed to 0.4 opacity; generating rows at full opacity; spinning `circleNotchIcon` reuses the existing `progress-spin` keyframe; status text capped at 140px with ellipsis; 2px orange progress bar at the bottom of generating rows with smooth width transition; notice and missing-source styles for the no-data state

### PMTiles Output

- **Output format expansion** (`config/types.ts`, `config/parser.ts`): added `pmtiles` to `OutputFormat` type; new `PMTilesOutputParams` interface with `minzoom` (default 0), `maxzoom` (default 14), `layerName` (defaults to source ID), optional `bbox` for spatial subsetting, and optional `layers` array for source layer filtering; new optional `params` field on `OutputConfig` for format-specific options; `VALID_OUTPUT_FORMATS` updated to include `pmtiles`
- **PMTiles param validation** (`config/validators/outputs.ts`): new `validatePMTilesParams()` validates minzoom/maxzoom as integers 0-22 with minzoom <= maxzoom, layerName as non-empty string, bbox as 4-number array with valid west < east and south < north, and layers as non-empty string array; params rejected when format is not pmtiles; PMTiles sources now conditionally allowed when output format is pmtiles (previously rejected for all formats)
- **Tiling pipeline** (`db/pmtiles-writer.ts`): new module generates PMTiles v3 vector tile archives from DuckDB feature data; pipeline reads features via `getFeaturesAsGeoJSON()`, slices with `geojson-vt` (new dependency) per zoom level within data bounds via `bboxToTileRange()`, encodes MVT with `vt-pbf` (new dependency), compresses tiles with `fflate` gzip, and builds the PMTiles v3 binary archive with hand-rolled header serialization, varint-encoded directory, and gzip-compressed metadata; archive layout places root directory immediately after the 127-byte header for compatibility with the pmtiles library's 16KB initial read; directory entry offsets are relative to the tile data section; runs in the DuckDB Web Worker thread with per-zoom-level progress reporting
- **Worker RPC** (`db/worker-types.ts`, `db/worker.ts`, `db/client.ts`, `db.ts`): new `ExportAsPMTilesRequest` type with dataset ID and optional params; worker handler dynamically imports the tiling pipeline to keep the initial bundle small; client `exportAsPMTiles()` function returns `Promise<Uint8Array>` via zero-copy transfer; re-exported through the barrel
- **Output executor** (`config/output-executor.ts`): added `pmtiles` MIME type (`application/octet-stream`) and switch case calling `exportAsPMTiles(source, params)`; exhaustive `never` check at `default:` provides compile-time safety for missing formats
- **Generator round-trip** (`config/generator.ts`): outputs with non-empty `params` are serialized to YAML during config generation
- **CSS badge** (`styles/config-viewer.css`): orange format badge for PMTILES in the outputs dropdown, distinct from green (GeoJSON), blue (CSV), and purple (Parquet)

### PMTiles File Upload

- **Upload support** (`controls/upload-control.ts`, `data-actions/load-file.ts`): `.pmtiles` added to the file input accept attribute; uploaded PMTiles files bypass the DuckDB worker pipeline and load as vector tile sources using the pmtiles library's `FileSource` (reads via `File.slice()`) instead of blob URLs, avoiding browser limitations with Range requests on blob URLs; the `PMTiles` instance is registered directly with the protocol handler by key matching `FileSource.getKey()` (filename); metadata read via the pre-created instance, not a URL fetch
- **Protocol access** (`map.ts`): exported `pmtilesProtocol` instance so file upload can register `PMTiles` instances via `protocol.add()`
- **Metadata reader** (`loaders/pmtiles.ts`): `getPMTilesMetadata()` now accepts a `PMTiles` instance in addition to a URL string, used by the file upload path to read metadata without HTTP
- **Load options** (`data-actions/load-pmtiles.ts`): new optional `pmtilesInstance` field on `PMTilesLoadOptions` allows passing a pre-created instance for metadata reading

### Output Execution

- **Worker-side export functions** (`db/features.ts`): three new export functions run inside the DuckDB Web Worker; `exportAsGeoJSON()` reuses `getFeaturesAsGeoJSONString()` and encodes to `Uint8Array`; `exportAsCSV()` and `exportAsParquet()` flatten the JSON `properties` column into individual columns via `json_extract_string` with a dynamic column list derived from `getAllPropertyKeys()`, which scans all features for the complete set of distinct property keys (unlike `getPropertyKeys()` which samples one row); CSV uses `ST_AsText(geometry)` for WKT; Parquet uses `ST_AsWKB(geometry)` for GeoParquet-compatible WKB; both use DuckDB `COPY TO` with a temp table for parameterized dataset filtering, then `db.copyFileToBuffer()` to read the virtual file as a `Uint8Array` with cleanup via `DROP TABLE` and `db.dropFile()`; property keys with special characters are escaped in JSON paths and SQL identifiers
- **Worker RPCs** (`db/worker-types.ts`, `db/worker.ts`, `db/client.ts`, `db.ts`): new `ExportAsCSVRequest` and `ExportAsParquetRequest` request types added to the discriminated union; worker switch cases use `respondTransfer` for zero-copy buffer transfer to the main thread; GeoJSON export reuses the existing `GetFeaturesAsGeoJSONRequest` handler returning the raw buffer; client wrappers `exportAsGeoJSON()`, `exportAsCSV()`, and `exportAsParquet()` return `Promise<Uint8Array>`; re-exported through the `db.ts` barrel
- **Output executor** (`config/output-executor.ts`): replaced the stub with a real implementation that calls the worker RPCs per output, creates `Blob` objects with correct MIME types (`application/geo+json`, `text/csv`, `application/octet-stream`), generates blob URLs for download, and returns `OutputResult[]` with size and blob URL populated; outputs execute sequentially to avoid memory pressure; per-output errors do not stop other outputs from executing
- **Progress logging** (`db/worker.ts`): CSV and Parquet export switch cases log to ProgressControl via `postProgress()` with processing and success statuses, flowing through the existing worker event handler pipeline

### CSV Geometry Import

- **WKT geometry parsing** (`loaders/csv.ts`): CSV files with a geometry column containing WKT strings (e.g., `POLYGON ((...))`, `LINESTRING (...)`) are now auto-detected and parsed into GeoJSON FeatureCollections; supports all standard WKT types: Point, LineString, Polygon, MultiPoint, MultiLineString, MultiPolygon, and GeometryCollection; handles optional Z/M/ZM coordinate suffixes (extra dimensions dropped to match GeoJSON 2D); EMPTY geometries silently skipped; WKT coordinate order (X Y = lng lat) matches GeoJSON so no flipping needed; enables round-trip import of CSVs exported by the app's own `exportAsCSV()` which serializes geometry as WKT via `ST_AsText()`
- **GeoJSON geometry string parsing** (`loaders/csv.ts`): CSV files with a geometry column containing JSON-serialized GeoJSON geometry objects are also auto-detected; handles raw geometry strings (`{"type": "LineString", "coordinates": [...]}`) and Feature wrappers (`{"type": "Feature", "geometry": {...}}`); covers the format used by Vancouver Open Data and other OpenDataSoft portal CSV exports where the `Geom` column contains stringified GeoJSON
- **Geometry column auto-detection** (`loaders/csv.ts`): new `detectGeometryColumn()` checks known column name aliases (`geometry`, `geom`, `wkt`, `the_geom`, `shape`) case-insensitively, validates the first row's value as either WKT or GeoJSON, then falls back to scanning all columns; detection runs after explicit `geoColumn` config but before lat/lng detection, so a CSV with both a `geometry` WKT column and lat/lng property columns uses the WKT as authoritative geometry; previous `detectWktColumn` retained as deprecated alias
- **Tests** (`loaders/csv.test.ts`): 57 new tests covering WKT value detection, GeoJSON geometry string detection, unified geometry column detection, WKT parsing for all seven geometry types (including holes, multi-geometries, geometry collections, Z coordinates, negative coordinates, case insensitivity), GeoJSON geometry parsing (raw and Feature wrapper), and integration tests through `csvLoader.load()` including WKT round-trip, GeoJSON LineString/Polygon import, Feature wrapper extraction, geometry-over-lat/lng priority, and the Vancouver open data CSV format

### Demo and Landing Page

## v0.7.0 - 2026-03-21

### PMTiles Loading

- **Protocol handler** (`map.ts`): installed `pmtiles` npm package and registered the `pmtiles://` protocol with MapLibre before map creation; the protocol handler translates `pmtiles://` URLs into HTTP range requests against the archive, so MapLibre can fetch individual tiles on demand without downloading the entire file; registration runs once at startup and is safe even when no PMTiles datasets are loaded
- **Config and format support** (`loaders/types.ts`, `loaders/detect.ts`, `config/parser.ts`, `config/types.ts`, `config/validators/datasets.ts`, `config/validators/layers.ts`): added `pmtiles` to `ConfigFormat` and `DetectedFormat` types; `.pmtiles` URL extension auto-detected via `EXTENSION_MAP` and `SEGMENT_MAP`; new `sourceLayer` field on `DatasetConfig` specifies which vector tile layer to use from a multi-layer archive; new `source-layer` field on `LayerConfig` for explicit layer configs targeting vector sources; dataset validator rejects `latColumn`, `lngColumn`, `geoColumn`, `paginate`, and `crs` when format is `pmtiles` since these are meaningless for vector tile sources
- **Vector source path** (`layers/sources.ts`, `db/core.ts`, `db/datasets.ts`, `db/worker-types.ts`, `db/worker.ts`, `db/client.ts`): PMTiles datasets bypass the DuckDB/loader pipeline entirely; new `addVectorSource()` creates a MapLibre `vector` source with a `pmtiles://` URL; new `createMetadataOnlyDataset()` worker RPC inserts a row into the `datasets` table with `feature_count = 0` and `format = 'pmtiles'` for layer panel tracking, visibility, and color; non-destructive schema migration adds `format` and `source_layer` columns to the datasets table (no schema version bump, follows the `layer_order` migration pattern); `getDatasets()` and `getDatasetById()` include the new columns in returned objects
- **Layer creation** (`layers/layers.ts`): `addLayerFromConfig()` passes through `source-layer` from config to the MapLibre layer spec; new `addDefaultVectorLayers()` creates the standard fill/line/circle triplet with `source-layer` set on each; supports an optional `layerSuffix` parameter to generate unique layer IDs when creating default layers for multiple source layers from the same archive (e.g. `dataset-protomaps-roads-fill`, `dataset-protomaps-buildings-fill`)
- **Config loading pipeline** (`data-actions/load-pmtiles.ts`, `data-actions/load-config.ts`): new `loadPMTilesDataset()` orchestrator reads the PMTiles header via a single range request to determine tile type, available vector layers, zoom range, and bounds; validates tile type is MVT (rejects raster with a clear error); resolves source layers automatically from header metadata; single-layer archives use that layer, multi-layer archives create default layers for all vector layers; explicit `sourceLayer` in config validated against the archive's layer list with available names shown in the error; creates the vector source, metadata-only DB entry, default layers with popup/hover handlers, and fits bounds from metadata; integrated into `loadDatasetsFromConfig()` as an early branch before the OPFS restore check and worker pipeline, with detection via `format: pmtiles` or `.pmtiles` URL extension
- **URL loading** (`data-actions/load-url.ts`): DataControl URL input detects PMTiles URLs (by format option or `.pmtiles` extension) and routes to `loadPMTilesDataset()` before any worker involvement, bypassing the fetch size check and DuckDB pipeline entirely; a 125GB worldwide tile archive loads instantly because only the header is fetched up front
- **PMTiles metadata reader** (`loaders/pmtiles.ts`): new main-thread module that reads the PMTiles archive header and TileJSON metadata via the `PMTiles` class from the `pmtiles` package; extracts vector layer names from `vector_layers` metadata, min/max zoom from header, geographic bounds, and tile type (MVT vs raster); used by the loading orchestrator to auto-detect source layers and validate tile format
- **OPFS session restore** (`data-actions/load-config.ts`, `map.ts`): PMTiles datasets persisted in OPFS are restored by re-adding the vector source from the stored `source_url` and recreating default layers from the stored `source_layer`, rather than attempting to materialize GeoJSON from DuckDB (which would return zero features); same restore logic added to `restoreNonConfigDatasets()` for manually loaded PMTiles datasets
- **Layer panel guards** (`controls/layer-control.ts`, `layer-actions/layer-row.ts`, `config/generator.ts`): PMTiles datasets show "PMTiles" instead of a feature count in the layer panel; "Export as GeoJSON" context menu item hidden when `dataset.format === 'pmtiles'`; PMTiles dataset IDs filtered out of the array passed to `createExecutionPlan()` so they cannot be used as operation inputs; config generator emits `format` and `sourceLayer` fields for PMTiles datasets
- **PMTiles example** (`examples/registry.ts`, `public/examples/configs/pmtiles.yaml`): new example page showing the Protomaps worldwide vector basemap with all nine source layers (earth, water, landcover, landuse, buildings, boundaries, roads, places, POIs) styled independently against the dark basemap using explicit layer configs with expressions for zoom-dependent road widths and label sizes

### Multi-layer Panel Entries

- **Sub-layer dataset convention** (`layers/sources.ts`, `layers/layers.ts`, `layers/index.ts`): PMTiles archives with multiple vector tile layers now create one DuckDB dataset entry per source layer using a `{parentId}/{sourceLayer}` ID convention (e.g., `protomaps/roads`, `protomaps/buildings`); `getSourceId()` extracts the parent ID from the convention so all sub-entries resolve to the shared MapLibre vector source; new `getLayersForDataset()` function scopes layer discovery by filtering on the MapLibre `source-layer` property, ensuring visibility, color, style, delete, and reorder operations target only the correct sub-layer's MapLibre layers
- **Per-layer panel entries** (`data-actions/load-pmtiles.ts`): multi-layer PMTiles archives auto-detect all source layers from the header and populate the layer control panel with separate rows per source layer; each entry gets an independent color from a nine-color palette, its own visibility toggle, style controls, and delete action; single-layer archives continue to create a single entry; when explicit `layers:` config exists, sub-entries are still created for panel visibility but auto-generated MapLibre layers are skipped (config layers handle rendering); new `resolveSubLayerColor()` extracts the primary color from matching config layer paint properties so panel rows reflect config-defined colors
- **Sub-dataset cleanup** (`db/datasets.ts`, `db/worker-types.ts`, `db/worker.ts`, `db/client.ts`, `db.ts`): new `deleteSubDatasets()` worker RPC deletes all dataset entries matching `{parentId}/%` before creating new sub-entries, handling reload and reconfig cleanly
- **Scoped layer operations** (`layer-actions/visibility.ts`, `layer-actions/color.ts`, `layer-actions/style.ts`): visibility toggle, color picker, and style sliders now use `getLayersForDataset()` instead of source-based lookup, correctly targeting only the sub-layer's MapLibre layers rather than all layers on the shared source
- **Shared source safety** (`layer-actions/delete.ts`): deleting a PMTiles sub-layer removes only that sub-layer's MapLibre layers; the shared vector source is only removed when no layers remain on it, so sibling sub-layers are unaffected
- **PMTiles style view** (`layer-actions/style.ts`): geometry presence detection returns all three types (fill, line, circle) for PMTiles datasets since DuckDB has no features to query; labels section hidden for PMTiles (no property keys in DuckDB)
- **OPFS restore** (`map.ts`): sub-layer entries that share a vector source skip duplicate `addVectorSource` calls; parent ID extracted from the dataset ID convention for correct MapLibre layer naming; config-covered sub-layers filtered from the manual restore path to avoid conflicts with the config pipeline

### PMTiles Sublayer Rename

- **Display-name rename** (`controls/layer-control.ts`, `layer-actions/style.ts`): PMTiles sublayers can now be renamed via the layer panel; since PMTiles dataset IDs encode the parent/sourceLayer relationship (`{parentId}/{sourceLayer}`), rename updates only the display name in DuckDB via `updateDatasetName()` rather than the full ID-based rename flow that rebuilds GeoJSON sources; MapLibre layers, sources, and layer IDs remain unchanged
- **Hover label update** (`controls/popup.ts`): new `updateHoverLabel()` function updates the tooltip display label for existing hover registry entries in-place, so renamed PMTiles sublayers show the new name in hover tooltips without removing and re-registering handlers
- **OPFS persistence** (`data-actions/load-pmtiles.ts`): when the config pipeline re-runs on page refresh, existing sublayer entries are read from DuckDB before `deleteSubDatasets()` wipes them; preserved user-modified names, colors, and styles are applied as overrides when recreating sublayer entries, so renames and color changes survive OPFS session restores
- **Paint restoration for all datasets** (`layers/layers.ts`, `map.ts`): `restoreStoredPaint()` no longer skips config-defined layers or PMTiles datasets; user color and style overrides are now applied to all layers using the existing expression-skip guard (`!Array.isArray`) to protect expression-driven paint properties, matching the behavior of the runtime `updateLayerColor()` function

### Bug Fixes

- **PMTiles config layer style not reflected in panel** (`data-actions/load-pmtiles.ts`): when explicit `layers:` config specified paint properties like `fill-opacity: 0.4` for PMTiles source layers, the style panel always showed default values (e.g., 0.2) because the DuckDB metadata entries stored `DEFAULT_STYLE` without incorporating config layer paint; colors worked correctly because `resolveSubLayerColor()` already extracted them; added `resolveSubLayerStyle()` (same pattern) to extract simple numeric paint values (opacity, width, radius) from matching config layers and merge them into the stored style before `createMetadataDataset()`, applied to both multi-layer sub-entries and single-layer entries
- **Config layer paint color not synced to dataset panel** (`map.ts`): datasets with explicit `layers:` config paint colors (e.g., `fill-color: "#f59e0b"`) showed the default `#3388ff` in the layer panel swatch and had their config paint overwritten on OPFS restore; root cause was that `executeLayersFromConfig()` applied paint to MapLibre layers but never wrote the color back to the DuckDB dataset entry, so `restoreStoredPaint()` used the default; PMTiles datasets were unaffected because `resolveSubLayerColor()` already handled this at load time; fixed by reading the actual MapLibre paint color via `getDisplayColor()` after layer creation and syncing it to DuckDB for any dataset still at the default color
- **Config layer opacity and style not applied** (`map.ts`, `data-actions/load-pmtiles.ts`): non-PMTile datasets with explicit `layers:` config paint properties (e.g., `fill-opacity: 0.8`, `line-width: 3`) rendered with default values instead of config-specified values; the style panel also showed defaults; root cause was the same as the color sync issue: `executeLayersFromConfig()` applied paint to MapLibre layers but `restoreStoredPaint()` then overwrote them with `DEFAULT_STYLE` from DuckDB because style properties were never synced back; PMTiles datasets were unaffected because `resolveSubLayerStyle()` already handled this at load time; fixed by reusing the exported `resolveSubLayerStyle()` to extract simple numeric paint values from config layers and sync them to DuckDB before `restoreStoredPaint()` runs
- **Clear Session not working on first try** (`db/worker.ts`): clicking "Clear Session" in StorageControl appeared to do nothing -- data reappeared after the page reload; root cause was `workerClearOPFS()` relying solely on deleting the OPFS file via `removeEntry()`, which silently failed when DuckDB's internal worker had not yet released the OPFS access handle after `database.terminate()`; fixed by flushing all table data (features, datasets, operations, meta) and checkpointing to OPFS before closing the database, so the persisted file is empty even if deletion fails; also added WAL file cleanup and a progress message so the UI shows feedback during the clear
- **Progress control not showing current operation** (`controls/progress-control.ts`): the minimized progress icon would skip showing operation progress on configs with both datasets and operations; root cause was a stale-message guard that unconditionally filtered all `processing` status updates when `scheduleIdle()` was active, including legitimate new operations starting after dataset loading; fixed by scoping the filter to only suppress messages for the same operation that just completed, allowing different operations to cancel idle and update the display

### PMTiles Extraction

- **Extraction pipeline** (`db/pmtiles-reader.ts`): new module extracts features from remote PMTiles archives via HTTP range requests; reads tile index and metadata to validate MVT tile type and available source layers; computes tile range from a user-specified bounding box and zoom level using `bboxToTileRange()`; fetches tiles with bounded concurrency (6 parallel requests, 10,000 tile cap); decodes MVT features using `@mapbox/vector-tile` and `pbf`; clips decoded geometries to exact tile bounds using Sutherland-Hodgman (polygons) and Liang-Barsky (lines) to eliminate buffer overhang from neighboring tiles; deduplicates across tile boundaries using centroid + properties hash keys rounded to ~1.1m precision; `extractPMTilesAndRebuild()` orchestrator chains extraction into the existing tiling pipeline, dynamically importing `generatePMTiles` or `generateMultiLayerPMTiles` based on layer count, with progress remapped across extraction (0-0.6) and re-tiling (0.62-0.98) stages
- **Source existence bypass for extraction** (`config/output-executor.ts`): `checkSourcesExist()` skips PMTiles extraction outputs (those with `extractZoom` in params) since they resolve source URLs from config datasets rather than DuckDB; `executeOutputs()` accepts an optional `datasets` parameter and resolves source URLs directly from the parsed config for extraction outputs, removing the DuckDB dependency for PMTiles-to-PMTiles workflows
- **Config control wiring** (`controls/config-control.ts`): `executeOutputs()` call updated to pass `config.datasets` as the third argument so extraction outputs can resolve source URLs
- **Extraction params validation** (`config/validators/outputs.ts`): PMTiles sources now require `extractZoom` and `bbox` params (cannot extract a full archive); validator updated to enforce both fields when the source is a PMTiles dataset
- **Test update** (`config/validators/outputs.test.ts`): PMTiles source acceptance test updated to include required extraction params
- **PMTiles example** (`public/examples/configs/pmtiles.yaml`): updated to use self-hosted Lower Mainland extract on R2 with Vancouver viewport (`fitBounds: false`) instead of the full Protomaps worldwide archive

### PMTiles Config Generation

- **Generate config support** (`config/generator.ts`): config generation now handles PMTiles multi-layer archives correctly; previously, each DuckDB sub-entry (e.g., `protomaps/water`, `protomaps/roads`) was emitted as a separate dataset with its own URL, producing invalid configs where explicit layers referenced `source: protomaps` but no dataset with that ID existed; the generator now collapses sub-entries into a single parent dataset entry and emits all source layers as explicit layer configs with the `source-layer` property; `extractLayerConfigs()` reads the MapLibre `source-layer` property from vector tile layers and includes it in generated output; auto-generated default layers with `source-layer` are no longer skipped (they were previously filtered by the flat-paint check that only applies to GeoJSON layers); coverage tracking uses the logical parent source ID rather than the incorrectly-parsed default layer ID; auto-generated layer IDs are cleaned up by stripping the `dataset-{source}-` prefix (e.g., `dataset-protomaps-water-fill` becomes `water-fill`)
- **Layer display name in config** (`config/types.ts`, `config/generator.ts`, `data-actions/load-pmtiles.ts`): added optional `name` field to `LayerConfig` for specifying layer display names in the panel; the generator detects user-renamed PMTiles sub-layers by comparing the DuckDB name against the default `"{parent} - {sourceLayer}"` pattern and emits the custom name on the layer config; on config load, `loadPMTilesDataset()` reads the `name` field from matching config layers and applies it when creating DuckDB sub-entries, so renamed layers round-trip through generate and reload
- **PMTiles layer order on config load** (`map.ts`): config-defined layer order was not preserved for PMTiles datasets after initial load; the `setLayerOrders` block mapped config layers to DuckDB dataset entries by `source` ID, but all PMTiles layers share the same source (e.g., `protomaps`), so all sub-entries received the same position and fell back to arbitrary DuckDB creation order; `resyncLayerOrder()` then overwrote the correct config layer order with this scrambled order; fixed by also indexing config layers by `{source}/{source-layer}`, so each sub-entry (e.g., `protomaps/water`) maps to its specific config layer position
- **Legend not grouping PMTiles sublayers** (`controls/legend-control.ts`): legend collapsed all PMTiles sublayers into a single entry because `buildLegendGroups()` grouped MapLibre layers by source, and all sublayers share one vector source; fixed by grouping on `source + source-layer` so each sublayer gets its own legend group with its correct name and color swatch; non-PMTiles datasets unaffected (no `source-layer`, falls back to source-only grouping)
- **Legend not updating on PMTiles sublayer rename** (`controls/layer-control.ts`, `map.ts`): renaming a PMTiles sublayer updated the display name in DuckDB and hover tooltips but the legend still showed the old name; root cause was that PMTiles renames are display-name only (no MapLibre layer ID changes), so no `styledata` event fires to trigger a legend rebuild; fixed by wiring the LegendControl into the LayerToggleControl and calling `refresh()` after a successful PMTiles rename
